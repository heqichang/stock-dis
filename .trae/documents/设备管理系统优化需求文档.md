# 设备管理系统优化需求文档

## 1. 文档说明

本文档基于现有设备管理系统（Win32 C++17 + SQLite），针对四项优化需求进行详细拆解，涵盖需求描述、现状分析、目标方案、技术设计及实现步骤。

---

## 2. 需求一：UI 字体与列表高度优化

### 2.1 需求描述

当前界面文字偏小，影响阅读体验；设备列表行高不足，数据展示不够舒适。需要整体放大字体、加高列表行高。

### 2.2 现状分析

| 位置 | 当前实现 | 问题 |
|------|---------|------|
| UIHelper 控件创建 | 使用 Win32 默认系统字体（约 9pt/12px） | 所有按钮、标签、输入框文字偏小 |
| DeviceDetailPage 自绘 | 标题 24px、标签 14px、值 14px、识别码 16px | 详情页字体尚可，但与控件字体不统一 |
| DeviceListPage 列表 | ListView 使用默认行高 | 行高偏矮，数据拥挤 |
| DeviceEditPage 表单 | 标签和输入框使用默认字体 | 表单标签和输入文字偏小 |

### 2.3 目标方案

#### 2.3.1 全局字体调整

- 创建全局统一字体，替换 Win32 默认系统字体
- 字体方案：

| 用途 | 字体名称 | 字号 | 粗细 |
|------|---------|------|------|
| 窗口标题/页面标题 | Microsoft YaHei | 18px (约13.5pt) | Bold |
| 标签文字（Label） | Microsoft YaHei | 16px (约12pt) | Normal |
| 输入框/按钮文字 | Microsoft YaHei | 16px (约12pt) | Normal |
| 列表文字 | Microsoft YaHei | 16px (约12pt) | Normal |
| 识别码/等宽文字 | Consolas | 16px (约12pt) | Normal |
| 提示/次要信息 | Microsoft YaHei | 14px (约10.5pt) | Normal |

#### 2.3.2 列表行高调整

- ListView 行高从默认（约 18px）增加至 **28px**
- 通过设置 ListView 的 `LVS_OWNERDRAWFIXED` 样式或使用 `ListView_SetItemSpacing` / 自绘方式实现
- 推荐方案：使用 `WM_MEASUREITEM` 消息处理，设置 `MEASUREITEMSTRUCT->itemHeight = 28`

#### 2.3.3 控件尺寸微调

配合字体放大，相应调整控件高度：

| 控件类型 | 当前高度 | 调整后高度 |
|---------|---------|-----------|
| 按钮 | 25-35px | 36-40px |
| 输入框 | 25-30px | 32-36px |
| 标签 | 25px | 28px |
| 下拉框 | 25px | 32px |
| 工具栏区域 | 55px | 65px |

### 2.4 涉及文件

| 文件 | 修改内容 |
|------|---------|
| `src/ui/UIHelper.h` | 新增全局字体创建/获取接口 |
| `src/ui/UIHelper.cpp` | 实现全局字体管理；所有 CreateXxx 函数创建控件后设置字体；ListView 行高设置 |
| `src/ui/DeviceListPage.cpp` | 调整控件布局坐标和尺寸 |
| `src/ui/DeviceEditPage.cpp` | 调整控件布局坐标和尺寸 |
| `src/ui/DeviceDetailPage.cpp` | 统一自绘字体与全局字体方案 |
| `src/ui/MainWindow.cpp` | 主窗口创建时初始化全局字体 |
| `src/ui/WindowBase.cpp` | WM_MEASUREITEM 处理（ListView 行高） |

### 2.5 实现步骤

1. 在 UIHelper 中新增 `InitFonts()` / `GetFont(FontType)` 接口，创建并缓存各类字体
2. 修改所有 `CreateXxx` 函数，创建控件后调用 `SendMessage(hWnd, WM_SETFONT, (WPARAM)GetFont(...), TRUE)`
3. 在 WindowBase 或 DeviceListPage 中处理 `WM_MEASUREITEM`，设置 ListView 行高为 28px
4. 调整 DeviceListPage 布局：工具栏区域高度从 55px → 65px，控件高度相应增大
5. 调整 DeviceEditPage 布局：标签/输入框高度和间距相应增大
6. 调整 DeviceDetailPage 自绘字体，使用全局字体替代局部 CreateFont
7. 程序退出时在 UIHelper 中释放字体资源

## 3. 需求二：设备参数表单化

### 3.1 需求描述

当前新增/编辑设备时，设备参数使用 JSON 文本框输入，对非技术用户不友好。需要改为结构化表单形式，字段类型可配置（文本、数字、日期等）。

### 3.2 现状分析

| 项目 | 当前实现 |
|------|---------|
| 存储格式 | `params` 字段存储 JSON 字符串，如 `{"品牌":"华为","型号":"MateBook X Pro"}` |
| 编辑方式 | DeviceEditPage 中使用多行文本框直接编辑 JSON |
| 用户体验 | 需要用户手动输入 JSON 格式，容易出错 |
| 字段定义 | 无字段配置，完全自由输入 |

### 3.3 目标方案

#### 3.3.1 参数字段配置表

新增数据库表 `param_fields`，用于存储可配置的设备参数字段定义：

```sql
CREATE TABLE IF NOT EXISTS param_fields (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    field_key TEXT NOT NULL UNIQUE,
    field_label TEXT NOT NULL,
    field_type TEXT NOT NULL CHECK(field_type IN ('text', 'number', 'date', 'select')),
    field_options TEXT DEFAULT '',
    sort_order INTEGER DEFAULT 0,
    is_required INTEGER DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_param_fields_key ON param_fields(field_key);
```

字段说明：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键自增 |
| field_key | TEXT | 字段标识（英文，如 brand、model），唯一 |
| field_label | TEXT | 显示名称（中文，如 品牌、型号） |
| field_type | TEXT | 字段类型：text（文本）、number（数字）、date（日期）、select（下拉选择） |
| field_options | TEXT | 下拉选项列表，JSON 数组格式，如 `["选项1","选项2","选项3"]`，仅 select 类型使用 |
| sort_order | INTEGER | 排序序号，控制表单中字段显示顺序 |
| is_required | INTEGER | 是否必填：0-否，1-是 |

#### 3.3.2 表单字段类型与控件映射

| 字段类型 | 控件 | 说明 |
|---------|------|------|
| text | 单行输入框（Edit） | 普通文本输入 |
| number | 数字输入框（Edit + ES_NUMBER） | 仅允许输入数字，可含小数点 |
| date | 日期选择器（DateTimePicker） | 弹出日历选择日期，格式 yyyy-MM-dd |
| select | 下拉选择框（ComboBox） | 从预定义选项中选择 |

#### 3.3.3 参数设置页面

新增"参数设置"页面，用于管理参数字段配置：

- 页面入口：在设备列表页工具栏增加"参数设置"按钮
- 功能：
  - 查看所有参数字段列表（字段名称、类型、排序、是否必填）
  - 新增参数字段：填写字段标识、显示名称、选择类型、设置选项（select 类型）、排序、是否必填
  - 编辑参数字段：修改字段属性
  - 删除参数字段：二次确认后删除
  - 调整排序：上移/下移按钮调整字段顺序

#### 3.3.4 设备编辑页表单化改造

将 DeviceEditPage 中的 JSON 文本框替换为动态生成的表单：

- 根据数据库 `param_fields` 表中的字段配置，动态生成表单控件
- 每个字段一行：左侧显示 field_label，右侧为对应类型的输入控件
- 必填字段在标签后标注 `*` 号
- 保存时将表单值序列化为 JSON 字符串存入 `params` 字段（保持与现有数据兼容）
- 编辑设备时，从 `params` JSON 中解析各字段值并回填到对应控件

#### 3.3.5 数据兼容性

- `params` 字段仍以 JSON 字符串存储在 `devices` 表中，**不改变现有表结构**
- JSON 的 key 对应 `param_fields.field_key`
- 已有设备的 params 数据不受影响，新增/编辑时按新字段配置解析
- 若 params 中存在未配置的字段数据，在表单中不显示，但保存时保留（不丢失）

#### 3.3.6 默认参数字段

系统首次启动时，自动插入以下默认参数字段配置：

| field_key | field_label | field_type | field_options | sort_order | is_required |
|-----------|-------------|------------|---------------|------------|-------------|
| brand | 品牌 | text | | 1 | 0 |
| model | 型号 | text | | 2 | 0 |
| purchase_date | 购入日期 | date | | 3 | 0 |
| price | 价格 | number | | 4 | 0 |
| location | 存放位置 | text | | 5 | 0 |

## 4. 需求三：导入导出功能修复（仅支持 Excel）

### 4.1 需求描述

当前导入导出功能未完整生效（Excel 导入未实现，JSON 导入导出不需要），需要修复并简化为仅支持 Excel 格式。

### 4.2 现状分析

| 功能 | 状态 | 问题 |
|------|------|------|
| 导出 JSON | ✅ 已实现 | 不再需要 |
| 导出 Excel | ✅ 已实现（libxlsxwriter） | 正常可用 |
| 导入 JSON | ✅ 已实现 | 不再需要 |
| 导入 Excel | ❌ 未实现 | `ImportFromExcel` 直接返回 `false` |
| 导出对话框 | 弹窗选择 JSON/Excel | 需简化为直接导出 Excel |
| 导入对话框 | 弹窗选择 JSON/Excel | 需简化为直接导入 Excel |

### 4.3 目标方案

#### 4.3.1 移除 JSON 导入导出

- 移除 `DataImportExport` 类中的 `ExportToJson` 和 `ImportFromJson` 方法
- 移除 `MainWindow::OnExport` / `OnImport` 中的格式选择对话框
- 导出直接弹出"另存为"对话框，仅支持 `.xlsx` 格式
- 导入直接弹出"打开文件"对话框，仅支持 `.xlsx` 格式

#### 4.3.2 实现 Excel 导入

引入 `libxlsxio` 库或使用替代方案读取 `.xlsx` 文件。可选方案：

| 方案 | 优点 | 缺点 |
|------|------|------|
| libxlsxio | C 语言库，与 libxlsxwriter 配套 | 需要额外引入依赖，依赖 libexpat |
| 手动解析 xlsx（ZIP + XML） | 无额外依赖 | 实现复杂，xlsx 是 ZIP 压缩的 XML |
| 使用 COM 自动化操作 Excel | Windows 原生支持，功能强大 | 依赖 Excel 安装，兼容性差 |
| 导出为 CSV 兼容格式 | 简单 | 不是真正的 xlsx 读取 |

**推荐方案**：引入 `libxlsxio` 库（读取端），与现有的 `libxlsxwriter`（写入端）配合使用，实现完整的 Excel 读写。

#### 4.3.3 Excel 导入数据映射

导入 Excel 时，按表头映射字段：

| Excel 列标题 | 映射字段 | 必须存在 |
|-------------|---------|---------|
| ID | device.id | 否（导入时忽略，自动生成） |
| 设备名称 | device.name | 是 |
| 识别码 | device.code | 否（为空则自动生成） |
| 状态 | device.status | 否（默认"使用中"） |
| 设备参数 | device.params | 否（默认"{}"） |
| 创建时间 | device.created_at | 否（默认当前时间） |
| 更新时间 | device.updated_at | 否（默认当前时间） |

导入规则：
- **设备名称**为必填列，缺失则跳过该行并记录错误
- **识别码**为空时自动按新规则生成（TSG 前缀 + 固定位数字）
- **识别码**已存在时跳过该行，避免重复
- 导入完成后显示结果摘要：成功 N 条，跳过 M 条，失败 K 条

#### 4.3.4 导出 Excel 优化

当前导出功能已可用，优化点：
- 设备参数列：将 JSON 中的各字段展开为可读格式（如 `品牌:华为, 型号:MateBook X Pro`），而非原始 JSON 字符串
- 列宽自适应优化

## 5. 需求四：识别码格式变更

### 5.1 需求描述

将设备识别码从当前的随机字母数字组合改为 `TSG` 前缀 + 固定位数字的格式。

### 5.2 现状分析

| 项目 | 当前实现 |
|------|---------|
| 格式 | 10 位随机大写字母+数字（如 `A3K9F2X7M1`） |
| 生成方式 | `GenerateUniqueCode()` 使用 `std::mt19937` 随机生成 |
| 唯一性保证 | 生成后查询数据库，重复则重新生成 |
| 位置 | `src/core/common.cpp` |

### 5.3 目标方案

#### 5.3.1 识别码格式

```
TSG + 6位数字
```

示例：`TSG000001`、`TSG000123`、`TSG999999`

格式说明：
- 前缀：固定 `TSG`（3 位）
- 数字部分：6 位，不足补零，范围 `000001` ~ `999999`
- 总长度：9 位
- 可容纳设备数量：999,999 台

#### 5.3.2 生成策略

采用**数据库自增序列**方式，确保唯一且有序：

1. 新增数据库表 `code_sequence`，存储当前序列值：

```sql
CREATE TABLE IF NOT EXISTS code_sequence (
    id INTEGER PRIMARY KEY CHECK(id = 1),
    current_value INTEGER NOT NULL DEFAULT 0
);

INSERT OR IGNORE INTO code_sequence (id, current_value) VALUES (1, 0);
```

2. 生成识别码流程：
   - 读取 `code_sequence.current_value`
   - `current_value += 1`
   - 格式化为 `TSG` + 6位补零数字
   - 更新 `code_sequence.current_value`
   - 在事务中执行，保证并发安全

3. 此方式的优势：
   - **唯一性**：由数据库序列保证，无需重试
   - **有序性**：识别码按创建顺序递增，便于管理和查找
   - **可读性**：格式统一，一眼可识别为公司设备编码
   - **性能**：无需循环检查重复

#### 5.3.3 兼容性处理

- 现有数据库中已有设备的识别码保持不变（旧格式数据不受影响）
- 新生成的识别码使用新格式
- 扫码查询、搜索功能同时兼容新旧格式
- 首次启动时，若 `code_sequence` 表不存在则创建，并扫描现有设备中 TSG 格式的识别码，将 `current_value` 初始化为最大值，避免重复

#### 5.3.4 识别码手动输入

- 新增设备时，识别码由系统自动生成，**不可手动修改**
- 编辑设备时，识别码不可修改（保持现有逻辑）

### 5.4 涉及文件

| 文件 | 修改内容 |
|------|---------|
| `src/core/common.h` | 修改 `GenerateUniqueCode()` 声明 |
| `src/core/common.cpp` | 重写 `GenerateUniqueCode()` 为 TSG + 数字格式 |
| `src/db/Database.cpp` | 新增 `code_sequence` 表创建和初始化 |
| `src/db/DeviceRepository.h` | 新增序列相关方法声明 |
| `src/db/DeviceRepository.cpp` | 实现 `GetNextCodeSequence()` 方法 |
| `src/ui/DeviceEditPage.cpp` | 识别码输入框始终只读，移除手动输入能力 |

### 5.5 实现步骤

1. 在 Database::CreateTables() 中新增 `code_sequence` 表创建 SQL
2. 在 DeviceRepository 中实现 `GetNextCodeSequence()` 方法（事务内读取+更新序列）
3. 重写 `GenerateUniqueCode()` 函数，调用 `GetNextCodeSequence()` 并格式化为 `TSGxxxxxx`
4. 修改 DeviceEditPage，确保识别码始终由系统自动生成
5. 添加首次启动兼容逻辑：扫描现有 TSG 格式识别码，初始化序列值
6. 移除 DeviceEditPage 中 `GenerateUniqueCode` 的循环重试逻辑（新方案无需重试）


| 阶段 | 需求 | 原因 |

| 第四阶段 | 需求三：导入导出修复 | 依赖新识别码格式，需要引入新库 |
